1. Which Firebase node?
onboardRequests/ (root-level node in the taxilatest Realtime DB).

Each record is keyed by a generated request ID like OB2605045G12I or REG-..., and stores one company application.

For reference, the related nodes used downstream are:

superClients/{cid} — created after approval (target of the approve flow)
companySettings/{cid} — per-company config also written on approval
But signups / new registrations are read only from onboardRequests/.

4. How is Firebase authenticated?
The Firebase Realtime Database legacy database secret — env var FIREBASE_DB_SECRET — passed as a query-string ?auth=<secret> on every HTTPS request to taxilatest.firebaseio.com.

There is no Admin SDK service account in use, and no user idToken flow for these reads. Specifically, from src/firebase.ts:

Line 4: export const DB_SECRET = process.env.FIREBASE_DB_SECRET || '';
Line 14: every fbRead / fbWrite builds the URL with 'auth=' + encodeURIComponent(DB_SECRET)
Lines 57, 61: an alternate helper does the same: ${DB_BASE_STRIPE}/${path}.json?auth=${process.env.FIREBASE_DB_SECRET}
This means the server has full-admin read/write on the Realtime DB regardless of Firebase Rules. The Firebase Web API key (line 6) is only used for Firebase Auth operations (creating users, password resets, lookups via identitytoolkit.googleapis.com) — not for DB reads.

Note on naming: your question asked about BW_FIREBASE_SECRET — that env var name is not used anywhere in this codebase. The actual variable name is FIREBASE_DB_SECRET, and it's already set in your Replit Secrets. BW_ADMIN_KEY (the one we set earlier) is a separate thing — it's for talking to the external Registration Replit's HTTP API, not Firebase.

Frontend caveat: the browser-side reads in Home.aspx (_fbGet('onboardRequests')) and the direct db.ref(...) writes in SA-Onboard.aspx use the client-side Firebase SDK — those are governed by your Firebase Rules, signed in as whichever Firebase Auth user is logged into the SA portal (or anonymously, depending on how the SDK is initialized). They are not using the DB secret. So your Firebase Rules must allow read/write on onboardRequests/ for super-admin users for the frontend portions to work — the backend will always work because it bypasses rules with the DB secret.
