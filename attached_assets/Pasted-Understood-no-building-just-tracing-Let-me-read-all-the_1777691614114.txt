The bug chain — 3 places where it breaks:
Bug #1 — _resolveAcceptance checks live data (already identified)

If the driver's trip-end Firebase event arrives before _resolveAcceptance runs (both fire within milliseconds of each other), driverdatarealx already shows Available → _driverIsBusy = false → takes Available path → convertstatus(Assigned) instead of [QueueJob] → job becomes Assigned (not Queued) → [GetQueuedJobs] returns 0 → stale path fires.

Bug #2 — NEW: [QueueJob] client ignores server rejection

Server handler (line 4029):

else if (_qJob.BookingStatus !== 'Offered' && _qJob.BookingStatus !== 'Pending' && _qJob.BookingStatus !== 'No One') {

Client _resolveAcceptance success callback (line 7928):

The success callback never reads resp.d — it never checks for ok: false. If [QueueJob] is rejected (e.g. the job was already changed to Assigned by Bug #1's race before the HTTP round-trip), the client logs "queued" and sets up the recall watcher as if it succeeded. _driverQueueMap[D001] stays set. When driver goes Available, [GetQueuedJobs] returns 0 → stale path fires.

Bug #3 — NEW: Server has no Queued → Offered guard

When the stale path fires and calls acknowledgemethodx, it calls DataProcessor [changeriddestatusforoffer] with ridestatus=Offered. The DataProcessor's guards only protect:

const isAccepted = currentStatus === 'Assigned' || currentStatus === 'Active' || currentStatus === 'Picking';

Queued is NOT in the protected set. So if the job actually IS Queued (Bug #1 did not occur, job correctly queued), but [GetQueuedJobs] still returned 0 for some reason, the stale path's acknowledgemethodx call will change the job from Queued → Offered, corrupting the queue assignment entirely. The job then appears in the Offer tab.
